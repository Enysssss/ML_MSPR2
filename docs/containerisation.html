1 Vue d'ensemble

L'API HealthAI Coach est packagée dans un conteneur Docker autonome : une seule image contient le serveur FastAPI, le modèle ML pré-entraîné et toutes les dépendances Python. Elle est déployée sur Railway sans configuration système particulière.

2 Dockerfile expliqué

Dockerfile 
FROM python:3.11-slim
# Image légère (~50 MB). Contient uniquement Python 3.11 et les outils essentiels.

WORKDIR /app
# Tous les fichiers seront dans /app. Évite les conflits de chemins.

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
# On copie requirements.txt EN PREMIER. Docker met en cache ce layer :
# si le code change mais pas les dépendances, pip ne tourne pas à nouveau.

COPY . .
# Copie tout le projet, y compris ml/models/model.pkl (pré-entraîné).

ENV PYTHONDONTWRITEBYTECODE=1 PYTHONUNBUFFERED=1
# PYTHONUNBUFFERED : logs visibles en temps réel dans Railway (sans buffer).

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=10s --start-period=15s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
# --host 0.0.0.0 est obligatoire : sans ça, uvicorn n'écoute que sur localhost
# interne et le conteneur est inaccessible depuis l'extérieur.

Optimisation du cache Docker

L'ordre des instructions dans un Dockerfile a un impact direct sur la durée des builds. Docker invalide le cache d'un layer dès qu'un fichier copié a changé — tous les layers suivants sont recalculés.

LayerSe recalcule si…Durée
pip installrequirements.txt change~90s
COPY . .N'importe quel fichier change<5s

En plaçant pip install avant COPY . ., les dépendances sont recalculées seulement si requirements.txt change — pas à chaque modification de code.

3 Health Check

Qu'est-ce qu'un health check ?

C'est une vérification automatique que Docker lance à l'intérieur du conteneur à intervalle régulier. Si l'app ne répond plus, Docker marque le conteneur unhealthy — Railway peut alors le redémarrer automatiquement.

Dockerfile — HEALTHCHECK 
HEALTHCHECK --interval=30s --timeout=10s --start-period=15s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1
ParamètreValeurSignification
--interval30sVérifie toutes les 30 secondes
--timeout10sPas de réponse en 10s → échec
--start-period15sLaisse 15s au conteneur pour démarrer avant de commencer les checks
--retries33 échecs consécutifs → conteneur unhealthy

La route /health

app/main.py 
@app.get("/health")
def health():
    return {"status": "ok", "version": "2.0.0"}

Cette route répond instantanément sans accès à la base de données ni au modèle ML. Elle confirme simplement que le processus uvicorn est en vie et capable de traiter des requêtes.

États du conteneur

démarrage │ ▼ (15s de grâce — start-period) starting │ ▼ GET /health répond 200 ? ├── Oui ──▶ healthy ✅ (vérification toutes les 30s) │ └── Non (3× consécutifs) ──▶ unhealthy ❌ │ ▼ Railway redémarre le conteneur

4 Disponibilité de l'API

Chargement du modèle au démarrage

Le modèle ML est chargé une seule fois au démarrage via un pattern Singleton. L'instance est créée au niveau module dans main.py, donc au moment où uvicorn importe l'application.

app/service.py 
class FitnessService:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._model   = joblib.load(MODELS_DIR / "model.pkl")
            cls._instance._encoder = joblib.load(MODELS_DIR / "encoder.pkl")
        return cls._instance

# app/main.py — exécuté au démarrage d'uvicorn
service = FitnessService()  # ← modèle chargé ici, une seule fois
Conséquence : la première requête /recommend répond aussi vite que les suivantes. Il n'y a aucun "cold start" par requête — le modèle est déjà en mémoire.

Le start-period de 15s dans le HEALTHCHECK est largement suffisant.

5 Problème résolu — le training au build

L'ancien Dockerfile

Dockerfile — AVANT (problématique) 
# Ces deux lignes tournaient à chaque docker build
RUN python ml/data/generate_dataset.py && \
    python ml/src/training/train.py

Pourquoi c'était lent

Le script train.py compare 3 algorithmes avec une recherche d'hyperparamètres exhaustive :

ml/src/training/train.py 
RandomizedSearchCV(
    estimator=pipeline,
    param_distributions=param_grid,
    n_iter=50,   # 50 combinaisons testées par modèle
    cv=5,        # validation croisée 5 folds
    scoring="f1_macro",
)
Entraînements par modèle
250
50 combinaisons × 5 folds
Modèles comparés
3
Random Forest, Gradient Boosting, XGBoost
Entraînements totaux
750
À chaque docker build
Durée du build
~15 min
À chaque git push

Impact sur Railway

À chaque git push, Railway reconstruisait l'image Docker en entier. Le training retournait systématiquement, bloquant le déploiement pendant 15 minutes — même si le code ML n'avait pas changé.

Avant / Après

Avant
  • Build : ~15 min (training inclus)
  • Déploiement disponible : ~18 min
  • Training à chaque deploy, même inutile
  • Résultat du modèle non garanti (seed aléatoire)
  • Pas de HEALTHCHECK
Après
  • Build : ~2 min (pip install seulement)
  • Déploiement disponible : ~3 min
  • Modèle chargé depuis git, stable
  • Reproductible — même fichier commité
  • HEALTHCHECK actif sur /health

6 Pourquoi pré-entraîner le modèle

Le modèle ne change pas à chaque déploiement

Le modèle apprend sur des données synthétiques générées avec une seed fixe (seed=42). Ces données ne changent pas entre les déploiements — re-entraîner produirait exactement le même résultat pour un coût de 15 minutes.

Modèle sélectionné

Algorithme
XGBoost
F1-macro

0.84 ±0.008

Avantages du modèle commité dans git

AvantageExplication
ReproductibilitéLocal, Railway, CI — tous utilisent exactement le même modèle
Traçabilitégit log ml/models/model.pkl montre quand et pourquoi le modèle a changé
RapiditéBuilds de ~2 min au lieu de ~15 min
StabilitéAucun risque qu'un deploy change silencieusement les prédictions

Quand faut-il re-entraîner ?

Re-entraîner est nécessaire uniquement dans ces cas :

CasFichier concerné
Les données d'entraînement changentml/data/generate_dataset.py
L'algorithme ou les hyperparamètres changentml/src/training/train.py
Les features changentml/src/preprocessing/engineer.py
Un bug dans le preprocessing est corrigéml/src/preprocessing/cleaner.py
Procédure de re-entraînement 
# 1. En local, dans le venv
python ml/data/generate_dataset.py
python ml/src/training/train.py

# 2. Vérifier les métriques dans les logs (F1-macro, accuracy)

# 3. Committer les nouveaux modèles
git add ml/models/model.pkl ml/models/encoder.pkl
git commit -m "chore: re-entraînement modèle — [raison]"
git push

7 Cycle de vie du conteneur

git push origin main
Railway détecte le nouveau commit sur la branche main
docker build (~2 min)
pip install les dépendances · COPY du code et de model.pkl · aucun training
~2 minutes
docker run → uvicorn démarre
Import de app.main → FitnessService() → chargement de model.pkl + encoder.pkl
~3-4 secondes
start-period (15s)
Docker attend avant de commencer les health checks — le conteneur peut démarrer sans être immédiatement marqué unhealthy
HEALTHCHECK — GET /health → 200
Conteneur marqué healthy · Railway redirige le trafic vers ce conteneur
Health checks toutes les 30s
Surveillance continue · 3 échecs consécutifs → redémarrage automatique du conteneur

8 Variables d'environnement

À configurer dans les paramètres Railway du projet.

VariableObligatoireDescription
DATABASE_URL Oui URL PostgreSQL — postgresql://user:pass@host:5432/db
FIREBASE_CREDENTIALS_B64 Non Credentials Firebase encodés en base64 (pour les logs de prédictions)
FIREBASE_CREDENTIALS Non Chemin vers le fichier JSON Firebase (alternative au B64)
Sans DATABASE_URL, les routes /nutrition/meals et /sessions/exercises retournent des erreurs 500. Les routes /health et /nutrition/calories fonctionnent toujours. Sans Firebase, /logs/* retourne 503 mais le reste de l'API est intact.

9 Checklist de déploiement

Déploiement normal (pas de changement de modèle)

  • git push origin main
  • Vérifier les logs Railway — build ~2 min, démarrage ~15s
  • Tester GET /health → {"status": "ok"}

Déploiement avec nouveau modèle

  • Modifier les données ou le code ML si nécessaire
  • Activer le venv : source .venv/bin/activate
  • python ml/data/generate_dataset.py
  • python ml/src/training/train.py
  • Vérifier les métriques dans les logs (F1-macro ≥ 0.84, accuracy ≥ 86%)
  • git add ml/models/model.pkl ml/models/encoder.pkl
  • git commit -m "chore: re-entraînement modèle — [raison]"
  • git push origin main
  • Vérifier les logs Railway
  • Tester POST /recommend avec un cas de test connu
